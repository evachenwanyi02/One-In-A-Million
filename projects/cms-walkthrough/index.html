第一部分：五个核心概念

在打开任何一个文件之前，先理解五个概念。它们是整个项目的骨架，所有代码都是这五个概念的具体实现。

概念一：分层架构

分层架构就像一栋大楼——每一层各有分工

这个项目像一栋楼，从底到顶分成四层。每一层只负责一件事，不干涉其他层的工作。

上层依赖下层，下层不知道上层的存在。改动只在本层内传播。

为什么分层？因为每一层只管一件事，互不干扰。就像电线和水管不会走同一根管道——虽然它们都藏在墙壁后面。当你修改一个按钮的颜色，不应该影响 API 请求的逻辑。分层让这种隔离自然发生。

概念二：Provider 套娃模式

Provider 像俄罗斯套娃——外层包裹内层，内层拥有外层的所有能力

React 用 Provider（提供者）向整个应用广播全局数据。App.tsx 里，四个 Provider 从外到内层层嵌套，每一层都给内部的所有组件注入一种「能力」：

套娃结构：里面的组件能使用所有外层 Provider 的能力。顺序有讲究——AuthProvider 需要发 API，所以在 QueryClientProvider 里面。
为什么用 Provider 模式而不是全局变量？因为 Provider 是 React 的响应式机制，数据变化时所有消费者自动更新。全局变量变了，React 不知道，页面不会重新渲染。这是面试高频问题。

概念三：数据怎么从后端流到页面

数据从后端服务器出发，经过层层处理，最终变成屏幕上的像素

一行 const { data } = useLeads() 背后的完整旅程。你不需要手动管理任何中间步骤。

这就是声明式编程的核心思想：你不是在告诉计算机「先请求、再存储、再渲染」（命令式），而是在说「我需要 leads 数据，你帮我管」。React 和 TanStack Query 让你从「怎么做」的细节里解脱出来。

当你删除一个 lead 时，useMutation 发完请求后调用 invalidateQueries——告诉缓存「leads 的数据过期了」。TanStack Query 自动重新请求最新列表，你不需要手动刷新。

概念四：认证流程

认证系统就是应用的门卫——验证身份，放行合法用户

用户输入密码后发生了什么？

整个认证流程对页面组件完全透明——你不需要在每个 API 调用中手动传 token。

概念五：主题系统

主题系统：一处定义颜色，全站统一变化

「这个蓝色到底是什么蓝？」「元素之间间距多少？」所有视觉相关的数值都集中在 theme.ts 里，通过 ThemeProvider 广播到全应用。

styled.d.ts 让 TypeScript 认识 theme 的结构，写 theme. 时编辑器弹出自动补全。

项目支持亮色和暗色两套主题（lightTheme / darkTheme），通过 ThemeModeContext 切换，状态保存在 localStorage 里，刷新页面后记住选择。

第三部分：代码语法详解

打开编辑器，看看每行代码到底在做什么

现在你知道了概念和文件地图，让我们打开关键文件，看里面的代码到底在做什么。大多数代码都是几个简单概念的组合——一旦你认识了基本积木块，再复杂的代码也只是排列组合。

src/main.tsx —— 九行代码的起点

import { StrictMode } from 'react'
import { createRoot } from 'react-dom/client'
import App from './App.tsx'

createRoot(document.getElementById('root')!).render(
  <StrictMode>
    <App />
  </StrictMode>,
)

import 就是「去仓库取货」。document.getElementById('root')! 里的 ! 是 TypeScript 的非空断言——你在保证这个值不是 null。StrictMode 是 React 的开发辅助，生产环境完全无效。

src/App.tsx —— ProtectedRoute 路由守卫

const ProtectedRoute: React.FC = () => {
  const { token, loading } = useAuth();
  if (loading) return null;
  return token ? <Outlet /> : <Navigate to="/login" replace />;
};

逻辑极简：有 token → 放行（渲染子路由插槽 Outlet），没有 → 跳转登录页。replace 表示「替换」历史记录，用户按返回键不会回到受保护页面。/login 和 /register 放在 ProtectedRoute 外面——否则就是「得先登录才能去登录页」的逻辑死循环。

const queryClient = new QueryClient({
  defaultOptions: {
    queries: { staleTime: 5 * 60 * 1000, retry: 1 },
  },
});

staleTime: 5 * 60 * 1000——数据 5 分钟内算「新鲜」，不重复请求。retry: 1——失败只重试一次。path="*" 是万用路由，匹配所有未定义路径并重定向到 dashboard。

src/api/client.ts —— 请求拦截器

// 请求拦截器：自动加 Token
client.interceptors.request.use((config) => {
  const token = localStorage.getItem('hermes_token');
  if (token && config.headers) {
    config.headers.Authorization = `Bearer ${token}`;
  }
  return config;
});

拦截器是 axios 的中间件机制——每个请求发出前自动跑一遍。Bearer 是 JWT 认证的标准格式。

// 响应拦截器：解包 + 处理 401
client.interceptors.response.use(
  (response) => {
    const body = response.data;
    if (body?.status === 'success' && 'data' in body) {
      response.data = body.data;  // 解包，调用方直接拿 data
    }
    return response;
  },
  (error) => {
    if (error.response?.status === 401) {
      localStorage.removeItem('hermes_token');
      window.location.href = '/login';  // Token 过期，跳回登录
    }
    return Promise.reject(error);
  }
);

后端返回 { status: 'success', data: {...} }，拦截器解包后调用方直接拿到 data，不用写 res.data.data。?. 是可选链——如果 error.response 是 undefined 就不继续访问 .status。

src/api/hooks.ts —— TanStack Query 的魔力

// 查询 Hook
export const useLeads = (params?: LeadListParams) =>
  useQuery({
    queryKey: ['leads', params],  // 缓存键，不同参数分开缓存
    queryFn: () => leadsApi.list(params).then(r => r.data),
  });

// 写操作 Hook
export const useCreateLead = () => {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: leadsApi.create,
    onSuccess: () => qc.invalidateQueries({ queryKey: ['leads'] }),
    // 不是删缓存，是标记「过期」，下次需要时自动重新请求
  });
};

queryKey 是缓存的键——相同 key 共享同一份缓存数据，params 也放进 key 让不同筛选条件分开缓存。invalidateQueries 标记缓存过期后，TanStack Query 自动刷新。

src/api/leads.ts —— TypeScript 工具类型

export type CreateLeadPayload = Omit<Lead, '_id' | 'createdAt' | 'updatedAt'>;
export type UpdateLeadPayload = Partial<CreateLeadPayload>;

Omit<Lead, ...> = Lead 去掉这三个字段（创建时由后端生成）。Partial<...> 让所有字段变可选（更新时可能只改其中几个）。不需要手动重新定义几乎相同的接口。

src/contexts/AuthContext.tsx —— Context 的完整实现

// 初始值从 localStorage 读取，刷新后不需要重新登录
const [user, setUser] = useState<User | null>(tokenStore.getUser());
const [token, setToken] = useState<string | null>(tokenStore.get());

// 组件挂载时：有 token 但没 user 信息就补回
useEffect(() => {
  if (token && !user) authApi.me().then(...).catch(...);
}, []);

// 登录函数
const login = useCallback(async (data: LoginPayload) => {
  setLoading(true);
  try {
    const res = await authApi.login(data);
    const { access_token, user: u } = res.data;  // user: u 是解构重命名
    tokenStore.set(access_token);
    setToken(access_token);
    setUser(u);
  } finally { setLoading(false); }  // finally 不管成功失败都执行
}, []);

关键语法：useCallback 记忆化函数避免不必要的重渲染。async/await 让异步代码读起来像同步。空依赖数组 [] 的 useEffect 只在组件挂载时跑一次。

// 防御性编程：在 Provider 外调用直接报错
export const useAuth = () => {
  const ctx = useContext(AuthContext);
  if (!ctx) throw new Error('useAuth must be used within AuthProvider');
  return ctx;
};

src/layouts/AppLayout.tsx —— CSS Grid 布局

const Shell = styled.div<{ $collapsed?: boolean }>`
  display: grid;
  grid-template-columns: ${({ $collapsed }) => $collapsed ? '61px' : '225px'} 1fr;
  min-height: 100vh;
  transition: grid-template-columns 0.3s ease;
`;

$collapsed 的 $ 前缀是 styled-components 的transient prop——不传递到真实 DOM。1fr = 剩余空间全占满。transition 让宽度变化有 0.3 秒动画。key={location.key} 让每次路由切换时触发淡入动画。

src/styles/GlobalStyles.ts —— 全局清零

export const GlobalStyles = createGlobalStyle`
  *, *::before, *::after {
    margin: 0; padding: 0; box-sizing: border-box;
  }
`;

box-sizing: border-box 让元素宽高包含 padding 和 border——否则默认只算内容区域，加 padding 会撑大元素，这是 CSS 最反直觉的默认行为之一。

src/components/Button —— 条件样式

const primaryStyles = css`
  background: ${({ theme }) => theme.colors.blue};
  color: #fff;
`;

const StyledButton = styled.button<{ $variant: string }>`
  ${({ $variant }) => $variant === 'primary' && primaryStyles}
`;

css 辅助函数把 CSS 封装成可复用片段。&& 短路求值：左边为 true 时返回右边的值（套用 CSS），否则不套用。

页面组件的共通模式

所有 CMS 页面都遵循同一个模式：import 依赖 → 定义 styled 样式组件 → 在函数组件里调用 useTranslation + API Hook → 用 useState 管理本地状态 → return JSX。

反复出现的语法：.map() 遍历数组渲染列表（每项都需要 key）、{'{'}isLoading && ...{'}'} 条件渲染、Object.entries() 把对象转成 [key, value] 数组。

React 的核心模型：数据从上往下流（props），事件从下往上冒（回调函数）。页面管理 state（筛选条件、当前页码），通过 props 传给子组件。子组件要改状态时调用父组件传下来的回调。

国际化 src/i18n/

i18n/index.ts 在应用启动时初始化 i18next，注册三种语言（繁中、简中、英文）。组件调用 useTranslation() 拿到 t 函数，用 t('nav.dashboard') 查翻译。切换语言时所有用 t() 的地方自动更新，不需要刷新页面。

实时事件流

sse.ts 里的 SSEClient 用 class 写成单例，通过浏览器原生 EventSource API 与后端建立长连接，接收 hermes_log、lead_update、email_sent 等事件。页面不在前台时弹浏览器通知。

Web Push（pushNotifications.ts + sw.js）更进一步——即使浏览器完全关闭，Service Worker 仍能在后台接收推送。SSE 需要页面打开，Push 不需要。

面试怎么讲这个项目

「我在一个三人团队里负责前端，项目是一个 B2B Lead Scraper CMS，用 React + TypeScript + Vite 搭建。我用 styled-components 搭建了完整的主题系统和 design tokens，实现了十多个共用组件；用 React Router 做了路由守卫和嵌套路由；用 TanStack Query 管理所有 API 状态，包括缓存和自动刷新；用 i18next 实现了三语支持，覆盖约 250 个翻译键；整合了 SSE 实时事件流和 Web Push 通知。后端用 NestJS 提供 RESTful API，统一响应格式，我在前端用 Axios 拦截器自动解包，JWT token 自动附加和 401 跳转都在拦截器里处理。」

你需要了解但不需要深入的后端知识

你不用会写 NestJS，但要能说清楚前后端接口：JWT 认证流程（怎么存 token、怎么自动带、过期怎么处理）、统一 API 响应格式（拦截器怎么解包）、CORS（跨域请求）。

业务方面：Lead 状态流转（new → pending → contacted）、SSE 事件类型、Email Queue 审批流程（pending → approved → sent 或 rejected）。核心原则：你不用知道后端怎么实现，但要能说出数据怎么流动、接口是什么。

写在最后

你在这段实习里学到的最重要的东西，不是 React 的语法（语法会变），不是 styled-components 的 API（库会换），而是读懂一个系统的能力——看到一堆文件时，知道怎么找到入口，怎么顺藤摸瓜追踪数据流向，怎么理解别人的设计决策。

面试时，不要试图背诵所有技术细节。展示你理解整体架构的能力——你知道数据从哪里来，经过哪些处理，最终怎么变成屏幕上的像素。这才是一个工程师最有价值的地方。
