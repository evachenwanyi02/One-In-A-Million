AI 偵探系統：架構全覽

數據層 → 組裝層 → Agent 層 → 運行層 ｜ 每層解決一個核心問題
先澄清：這裡的「智能體 Agent」不是「有手有腳」的那種
工具型智能體（能行動的）
  • 大模型 + 搜索 + 寫代碼 + 調接口
  • 能自主決定下一步做什麼
  • ← 你理解的智能體
角色型智能體（只會說話的）
  • 大模型 + 限定的知識 + 入戲指令
  • 只做一件事：用這個角色的身份說話
  • ← 本項目的嫌疑人智能體
智能體 ＝ 大模型 + 上下文約束（+ 可選的工具）。本項目的嫌疑人智能體不掛工具，只掛知識和人設。
第一層 數據層 — 案件怎麼存的 schema.ts + case-001.ts
每個案件是一份結構化數據檔案，用 Zod（格式校驗庫）約束，包含以下資料：
公開資料
  • 案件簡報 briefing
  • 時間線 timeline
  • 證物列表 evidence
  • 明線索 open — 玩家直接看到
  • 暗線索 hidden — 需觸發才解鎖
每個嫌疑人各自的資料
  • 姓名 / 性格 / 不在場證明
  • 秘密 secret、動機 motive
  • 知識陣列 knowledge[]
  • 劇情線 narrativeThreads[]
  • 態度演變 progressionGuidance
兇手獨有的資料
  • isCulprit = true
  • 撒謊策略 lyingStrategy
  • 要隱瞞的事 hide[]
  • 要編造的話 fabricate[]
  • 備選謊話 fallbackClaim v2
  • 放棄條件 abandonWhen v2
  • 只注入兇手自己的提示詞
改版新增的結構 NEW
  • helpfulInfo[] — 嫌疑人可以主動提供的有用信息，含觸發條件和偵探價值
  • naturalLeads[] — 自然引導線，在特定話題下聯想到其他嫌疑人/證物（不是「你去問 xxx」）
  • discoveredFacts[] — 偵探從各嫌疑人處蒐集到的已知事實（存在 GameState 裡）
  • narrativeThreads[] — 每條劇情線拆分：canReveal（可以說）/ mustHide（不能說）/ naturalConnection（自然聯想）
  • revealStages[] — 暗線索分階段揭示，每階段有獨立觸發條件和內容 v2
  • EmotionState — 四維情緒：pressure / trust / fatigue / guardedness（各 0-10） v2
  • MemorySummary — 對話歷史壓縮摘要，覆蓋式更新（每嫌疑人最多一條） v2
  • evidenceProgress — 每條證物目前解鎖到第幾階段（GameState） v2
三塊資料物理上都在同一個檔案裡，但組裝層會決定「發給大模型的時候只拿哪些」。
知識級別不是代碼邏輯（沒有 if/else 條件判斷），而是寫成自然語言指令讓大模型自行判斷回答語氣。
撒謊策略也是自然語言指令：告訴大模型「你要隱瞞 X、編造 Y」，大模型自行組織措辭。
第二層 組裝層 — 資訊隔離 information isolation 在這裡發生（核心） agent.ts buildSystemPrompt()
「組裝嫌疑人上下文」函數 buildSuspectContext(suspect) 做了什麼：
  • 輸入：一個嫌疑人物件（比如張浩）
  • 函數只讀張浩的知識陣列 → 不會去碰林美華或趙莉的資料
  • 檢查「是否為兇手」isCulprit → 如果是 true，追加撒謊策略段落
  • 如果是 false（無辜者），撒謊策略這段文字根本不會出現在提示詞裡
  • 渲染 narrativeThreads[] → 每條劇情線的「可以說 / 不能說 / 自然聯想」
  • 渲染 helpfulInfo[] → 「當話題涉及X時，你可以主動說出Y」 NEW
  • 渲染 naturalLeads[] → 「聊到X時，自然帶出Y」 NEW
  • 渲染 progressionGuidance → 前期/中期/後期/崩潰條件的態度演變
最終拼出的系統提示詞 system prompt 包含六個模組：
公開簡報所有人一樣
+
個人上下文只有自己的知識
+
情緒態勢v2
+
已知事實NEW
+
記憶摘要v2
+
行為規則輸出格式要求
→ 這就是「資訊隔離」：不是叫大模型「別說」，而是讓它「不知道」。
→ 組裝層的唯一職責：決定哪些資訊進入提示詞、哪些不進。控制輸入，而非控制輸出。
改版新增的提示詞規則 NEW
  • 兇手 = 冷靜的棋手：極其理智聰明，謊言嵌入大段真話中，面對矛盾先冷靜思考再回答
  • 無辜者 = 主動的證人：坦然交代、主動提供觀察、自然引出線索、有自己的推理和判斷
  • 絕對禁止：「你想想看」、反覆「不知道」、「去問 xxx」、歇斯底里、持續迴避
  • 回答長度可變：簡單問題 1-2 句直答；觸發劇情線 5-8 句展開；被追問矛盾可能只有短短一句辯解
  • 情緒注入：Clue Agent 評估的四維情緒狀態（pressure/trust/fatigue/guardedness）寫入 prompt，影響角色表現 v2
  • 跨嫌疑人信息：偵探從其他人處問到的事實（discoveredFacts）注入 prompt，讓角色知道偵探掌握了什麼
  • 記憶摘要：≥6 輪後自動壓縮歷史，摘要注入 system prompt + 最近 3 輪完整對話 v2
  • 動態欺騙：兇手的謊話根據 discoveredFacts 自動切換到備選版本 v2
第三層 Agent 層 — 數據怎麼餵給 LLM、LLM 怎麼用 agent.ts + clue-agent.ts + memory-agent.ts
我們的數據層有大量結構化資訊（knowledge、narrativeThreads、helpfulInfo、naturalLeads、lyingStrategy……），但大模型不是數據庫查詢引擎。它的工作方式是：你把一段文字塞給它，它根據文字回覆。所以 Agent 層的核心任務就是——把結構化數據翻譯成自然語言指令，拼成一份「角色劇本」，讓 LLM 按劇本演戲。
下面用 Agent → 能力單元 → 函數/數據 的結構來拆解。
Agent 的能力單元分三種：
· Prompt Module（靜態模組）：讀取靜態數據，組裝成一段 prompt 片段。沒有運行時邏輯，每次輸入相同則輸出相同。
· Dynamic Module（動態模組）：讀取運行時狀態（情緒、已知事實等），每輪產出不同的 prompt 片段。有 if/else 判斷，但最終產物仍是 prompt 文字。
· Skill（技能）：有獨立的執行單元——自己的 LLM 呼叫或決策邏輯，產出結構化數據並寫入系統狀態。這才是真正的「技能」。
好的能力單元的設計標準（貫穿下面每一個模組和技能的設計思路）：
① 數據邊界清晰：每個單元對應一塊獨立的數據結構，不會出現「這塊數據到底歸誰管」的模糊地帶
② 形成反饋迴路：最有生命力的單元不是注入一段靜態文字——它的輸出會變成下一輪的輸入，讓角色越來越「活」
③ 有降級路徑：LLM 掛了不能讓遊戲卡住，每個依賴 LLM 的單元都有 fallback
④ 控制輸入不控制輸出：不告訴 LLM「不要做什麼」，而是不給它做的材料——隔離信息而非限制行為
⑤ 向後兼容：所有新欄位 .optional() 或 .default()，舊案件不加新欄位也能正常運行
⑥ 共用 LLM 呼叫：判定邏輯高度重疊的單元合進同一次呼叫，不浪費 token
Agent ① Suspect Agent — 嫌疑人角色扮演 agent.ts · temperature 0.7 · maxTokens 600
用一句話概括：把案件數據變成角色劇本，讓 LLM 扮演一個有血有肉的嫌疑人。溫度 0.7 是因為角色扮演需要多樣性和創造力——同樣的問題問兩次，回答不應該一模一樣，就像真人不會每次用同樣的措辭。
Suspect Agent 的能力拆解：4 Prompt Modules + 2 Dynamic Modules + 1 Skill + 1 Fallback
分三層理解：Prompt Modules（靜態組裝）（1-4 + 6：讀取靜態案件數據，拼成 prompt 片段。其中 6 是條件式——只在 isCulprit 時注入）→ Dynamic Modules（動態組裝）（5 + 7：讀取運行時狀態 EmotionState/discoveredFacts，每輪產出不同的 prompt 片段）→ Skill（獨立執行）（8：有自己的 LLM 呼叫，產出 MemorySummary 寫入 GameState）。
Module 1 資訊隔離 — 「只讓你知道你該知道的」Prompt Module
每個嫌疑人的 prompt 裡只有自己的知識。張浩看不到趙莉知道的事，反之亦然。函數 buildSuspectContext(suspect) 只讀一個嫌疑人物件。每條 knowledge fact 帶知識級別：[親眼所見] → 語氣確定；[聽說] → 自動加「我聽人說」；[不知情] → 直接說「不知道」。
設計原則 ④：不是告訴 LLM「不要說」——是壓根不給它看。控制輸入而非控制輸出。
數據：suspect.knowledge[] · 函數：buildSuspectContext()
Module 2 劇情推進 — 「你的劇本是什麼、什麼時候演」Prompt Module
每個人有 3-5 條「劇情線」，每條是一個話題包。LLM 收到問題後先判斷「觸及了哪條劇情線」，然後按劇本展開。每條劇情線翻譯為三段指令：canReveal（可以說的）、mustHide（絕不能說）、naturalConnection（自然聯想），加上 triggerHints 告訴 LLM 進入條件。
舉例：偵探問「你那天跟他喝了什麼？」→ 觸及「那天晚上的紅酒」劇情線 → canReveal 告訴 LLM 可以說紅酒是老陳自己開的 → mustHide 阻止它提到下毒 → naturalConnection 讓它順帶提到趙莉。
設計原則 ①④：每條劇情線是獨立的數據單元（邊界清晰）；mustHide 依然是控制輸入——不把「怎麼說」的指令給它，它自然不會說。
數據：suspect.narrativeThreads[] · 函數：buildSuspectContext() 劇情線段
Module 3 主動情報 — 「有用的話，主動說出來」NEW Prompt Module
改版前無辜嫌疑人只會「被動回答」。改版後每個人帶一組 helpfulInfo[]，當話題涉及時會像回憶往事一樣主動說出來——讓無辜者從「路人」升級為「積極的破案助手」。每條帶觸發條件，翻譯成 prompt 指令。
舉例：偵探問林美華「張浩這人怎麼樣？」→ 她主動說「張浩說自己不太喝酒，但我在飯局上看他灌了半瓶白酒面不改色」——直接戳穿張浩「只喝了一口」的借口。
設計原則 ①：helpfulInfo 和 narrativeThreads 職責分開——劇情線管「怎麼回答」，helpfulInfo 管「主動補充什麼」。數據邊界清晰。
數據：suspect.helpfulInfo[] · 函數：buildSuspectContext() 有用信息段
Module 4 自然引導 — 「不說『去問 xxx』，但把你引過去」NEW Prompt Module
嫌疑人在敘述中用自然聯想把偵探引向其他嫌疑人或某條證物。每條 naturalLead 帶話題觸發條件和一段自然語氣的引導語。數據裡有 targetSuspectId 標記指向目標，但 LLM 不知道 ID——它只按自然語氣說話，引導效果由案件設計者掌控。
舉例：趙莉自然說「有個男人帶著公文包進了後門，九點半他的車好像還停在那」→ 偵探自己想到：張浩說九點走的，車怎麼還在？→ 去追問張浩。
設計原則 ④：不給 LLM 「targetSuspectId」這種 meta 信息，只給它自然語氣的台詞。LLM 不知道自己在「引導」——它只是在講述。
數據：suspect.naturalLeads[] · 函數：buildSuspectContext() 自然引導段
Dynamic 5 動態適應 — 「根據局勢變化調整表演」NEW Dynamic Module
融合三種動態數據，讓角色的表現隨對話推進而變化：
· progressionGuidance（態度演變）：四階段指令（前期坦然 → 中期謹慎 → 後期疲憊 → 崩潰條件），LLM 根據「第 N 次提問」自己判斷所處階段
· EmotionState（四維情緒 v2）：來自 Clue Agent 的上一輪評估。pressure/trust/fatigue/guardedness 四個維度，不同組合產生不同角色表現。壓力 ≥ 6 時帶身體語言提示
· discoveredFacts（偵探已知）：偵探從其他人處問到的信息，注入為「偵探可能已經知道的」段落
設計原則 ②：這是整個系統最核心的反饋迴路——Clue Agent 輸出情緒 → 注入下一輪 Suspect Agent → 影響回答 → Clue Agent 再次評估。第 1 輪和第 10 輪的 prompt 完全不同。
數據：progressionGuidance + EmotionState + discoveredFacts[] · 函數：buildEmotionContext() + buildDiscoveredFactsContext()
Module 6 精準欺騙 — 「兇手怎麼撒謊」（僅兇手）Prompt Module（條件式）
不是讓 LLM 自己決定怎麼撒謊——而是給它一份「欺騙劇本」。hide[] 規定隱瞞什麼，fabricate[] 規定在什麼話題上編什麼說辭，fallbackBehavior 規定被逼急了怎麼轉移話題。兇手定位為「冷靜的棋手」——謊言嵌在大段真話中。
設計原則 ④：無辜者不需要這個 Skill——因為組裝函數檢查 isCulprit，false 時 lyingStrategy 整段不出現在 prompt 裡。不是告訴 LLM「你不要撒謊」，而是壓根不給它撒謊的腳本。
數據：lyingStrategy { hide[], fabricate[], fallbackBehavior } · 函數：buildSuspectContext()（if isCulprit）
Dynamic 7 動態欺騙 — 「謊話被拆穿就換一個」（僅兇手）NEW v2 Dynamic Module
Skill 6 的進化：每條 fabricate 新增 fallbackClaim（備選謊話）和 abandonWhen（放棄條件）。組裝時掃描 discoveredFacts 是否命中 abandonWhen 的關鍵詞——命中就自動切換到備選謊話。判斷用字串匹配，不額外呼叫 LLM。
舉例：張浩主謊「我九點走了」，abandonWhen 含「車還在」。偵探從趙莉處得知「九點半車還停在那」→ discoveredFacts 命中 → prompt 自動切換：「哦對，我後來又折回去拿公文包了……」
設計原則 ②⑤：這也是反饋迴路的一部分——discoveredFacts 越多，兇手的 prompt 就越不同。而且 fallbackClaim 和 abandonWhen 都是 .optional()，不加就等於 Skill 6 的舊行為。
數據：fabricate[].fallbackClaim + abandonWhen · 函數：buildSuspectContext() 動態渲染
Skill 8 記憶摘要 — 「長對話不丟重點」NEW v2 Skill（獨立 LLM 呼叫）
≥6 輪後，完整歷史太長。用一次低溫度呼叫（temp 0.1）壓縮為三段結構化摘要：【嫌疑人聲稱的事實】+【偵探追問重點】+【已暴露的可疑之處】。摘要注入 system prompt，最近 3 輪保留為完整 chat history。
設計原則 ③：LLM 摘要失敗時，自動退回「簡易摘要」——從 highlights 和回答前 50 字拼成。不阻塞遊戲。這也是唯一一個產生額外 LLM 呼叫的 Skill（由獨立的 Memory Agent 執行）。
數據：GameState.memorySummaries[] · 函數：summarizeHistory() + buildSummarizedHistory() · Agent：memory-agent.ts
Suspect Agent 的輸入和輸出：
輸入 = 5 個 Module + 2 個 Dynamic Module 產生的 prompt 段落拼成的 system prompt +（Skill 8 的摘要或完整歷史）+ 這一輪的新問題
輸出 = 結構化 JSON（用 Zod 校驗，失敗時 fallback 提取）：
　· innerMonologue：內心獨白——LLM 先想「觸及了哪條劇情線？該說什麼、藏什麼、聯想什麼？」（玩家在揭曉階段才看到）
　· reply：說出口的話——長度自然變化，短則 1-2 句事實回答，長則 5-8 句故事展開
　· liedAbout[]：這句話裡撒了什麼謊（空 = 沒撒謊。復盤時用）
　· highlights[]：LLM 自己標出的關鍵短語（時間、地點、證物。前端高亮用）
主函數：interrogate(provider, caseData, suspectId, question, history, pressureCtx, discoveredFacts, memorySummary)
Agent ② Clue Agent — 線索觸發 & 情緒評估 clue-agent.ts · temperature 0.2 · maxTokens 300
用一句話概括：用 LLM 做「裁判」而不是「演員」。Suspect Agent 讓 LLM 扮演角色（創造性，溫度 0.7），Clue Agent 讓 LLM 做判定（需要穩定一致，溫度 0.2）。同一個 LLM，不同溫度 = 不同用途。Clue Agent 在 Suspect Agent 之後運行——先讓嫌疑人回答，再讓裁判判定這段對話的「後果」。
Clue Agent 的 4 個 Skill（全部是真正的 Skill — 有獨立的 LLM 呼叫，產出結構化數據）：
和 Suspect Agent 不同，Clue Agent 的每個能力單元都是真正的 Skill——它們有自己的 LLM 呼叫，產出的不是 prompt 文字而是結構化數據（unlockedIds、EmotionState、stageReveals），並且會寫入 GameState 改變系統狀態。分層：核心判定（1：語義觸發）→ 附加評估（2-3：情緒和階段推進，共享同一次 LLM 呼叫）→ 安全網（4：降級容錯）。Skill 1-3 在同一次 LLM 呼叫裡完成——這是設計原則 ⑥ 的極致體現。
Skill 1 語義觸發 — 「對話是否解鎖了暗線索？」
遊戲裡有些證物是「暗線索」——一開始不給玩家看，需要在對話中「觸發」才解鎖。這個 Skill 判斷每一輪對話是否觸發了某條暗線索。把偵探的問題、嫌疑人的回答、以及所有未解鎖暗線索的觸發條件描述一起塞給 LLM，問它「這段對話是否涉及了某條暗線索？」。
為什麼用 LLM 而不用代碼：代碼只能做關鍵詞匹配——對話裡出現「公文包」才觸發。但玩家可能問「他離開的時候手上拿的東西呢？」，沒有「公文包」三個字，語義上卻在問同一件事。LLM 能理解這種語義等價，代碼做不到。
舉例：張浩說「老陳喝酒快，一杯下去眨眼的事」→ LLM 判定這段話語義上涉及了「紅酒杯」相關話題 → 觸發暗線索 ev-7「紅酒杯指紋比對報告」→ 偵探面板上多出一條新證物。
設計原則 ④⑥：只給 LLM 觸發條件的「描述」，不給它「應不應該觸發」的判斷框架——控制輸入讓 LLM 自己決定。同時，語義觸發和情緒評估、階段推進共用同一次 LLM 呼叫，不浪費 token。
數據：evidence[].visibility === "hidden" 的暗線索清單 + triggerKeywords[] · 函數：checkCluesWithAgent() · 輸出：unlockedIds[]
Skill 2 多維情緒評估 — 「嫌疑人現在什麼狀態？」NEW v2
在同一次 LLM 呼叫中（跟線索觸發共用），評估嫌疑人的四維情緒狀態。不再是單一壓力等級，而是更細膩的多維模型。情緒結果回傳到下一輪 Suspect Agent 的 Dynamic Module 5（動態適應），影響角色的整體表現。
四維情緒模型 EmotionState（每個 0-10 分）：
· pressure（壓力）：被追問的緊迫感。0=毫無壓力 → 10=瀕臨崩潰
· trust（信任）：對偵探的配合意願。0=完全敵對 → 5=中立 → 10=完全配合
· fatigue（疲憊）：審問消耗。0=精神充沛 → 10=精疲力竭
· guardedness（戒備）：被觸及敏感話題的警覺。0=坦然 → 10=高度警惕
不同組合產生不同角色表現：高信任+低壓力→主動交代；低信任+高壓力→敷衍沉默；高疲憊+高戒備→簡短但措辭小心。壓力 ≥ 6 時額外生成身體語言提示（如「嫌疑人的手微微發抖」）。
向後兼容：保留舊的 PressureLevel（none/light/heavy/breakthrough）作為 pressure 數值的簡化映射。0-3→none, 4-5→light, 6-8→heavy, 9-10→breakthrough。
設計原則 ②⑤⑥：這是系統最關鍵的跨 Agent 反饋迴路——Clue Agent 輸出情緒 → Suspect Agent Dynamic 5 注入 → 影響回答 → Clue Agent 再次評估。同時向後兼容（PressureLevel 映射），且和 Skill 1/3 共享 LLM 呼叫。
數據：GameState.emotionStates { suspectId: EmotionState } · 函數：checkCluesWithAgent() + buildEmotionContext() · 輸出：emotion + emotionHint
Skill 3 分階段揭示 — 「越挖越深，一層一層揭開」NEW v2
原版暗線索是一次性開關——觸發就全部揭示。分階段揭示讓某些證物可以像剝洋蔥一樣，偵探每追問到一個新深度就揭示更多細節。每個階段有獨立的觸發條件，偵探需要步步深入才能看到完整真相。
怎麼做：證物新增可選欄位 revealStages[]，每個階段包含 stage（編號）、content（揭示內容）、triggerDescription（觸發條件）。Clue Agent 在同一次 LLM 呼叫中判斷已部分解鎖的證物是否觸發了下一階段。GameState.evidenceProgress 記錄每條證物目前解鎖到第幾階段，getRevealedContent() 和 getEvidenceDescription() 負責把階段進度轉換為前端可用的內容。
舉例：ev-6「法醫屍檢報告」拆為 3 階段：
· 階段 1：死因概述（毒殺） → 問到死因/毒物就觸發
· 階段 2：毒物細節（哪個杯子有毒） → 問到兩個杯子/下毒方式才觸發
· 階段 3：精確死亡時間 21:15-21:45 → 問到時間矛盾/張浩離開時間才觸發
沒有 revealStages 的證物 = 現有行為不變（一次全部揭示）。
設計原則 ①⑤⑥：每個 stage 是獨立的數據單元，觸發條件互不干擾（邊界清晰）。revealStages 為 .optional()——沒有它的證物完全不受影響（向後兼容）。階段判定和線索觸發、情緒評估共享同一次 LLM 呼叫。
數據：evidence[].revealStages[] + GameState.evidenceProgress · 函數：checkCluesWithAgent() + getRevealedContent() + getEvidenceDescription() · 輸出：stageReveals[]
Skill 4 降級容錯 — 「LLM 掛了也不影響遊戲」
LLM 呼叫可能失敗（網路超時、返回格式錯誤、服務不可用）。這個 Skill 確保 Clue Agent 即使在 LLM 不可用的情況下也能工作——自動降級到純關鍵詞匹配。try/catch 包住 LLM 呼叫，失敗時回退到 ClueManager（純代碼匹配器），用 evidence.triggerKeywords[] 和對話文字做字串比對。
降級的代價：語義觸發退化為關鍵詞觸發（準確度下降），情緒評估回傳全零（EmotionState 四維都是 0），分階段揭示停止推進。但遊戲不會卡住——這是底線。
設計原則 ③：這個 Skill 本身就是降級路徑。它不產生新能力，它保護前面 3 個 Skill 的所有能力不會因為 LLM 故障而讓遊戲崩潰。三層降級：LLM → ClueManager 關鍵詞匹配 → 空結果（ClueManager 也不可用時）。
數據：evidence[].triggerKeywords[] · 函數：ClueManager.checkTriggers()（fallback） · 輸出標記：usedLLM: false
Clue Agent 的輸入和輸出：
輸入 = 偵探的問題 + 嫌疑人的回答 + 未解鎖暗線索清單 + 可推進的分階段線索清單
輸出 = ClueAgentResult { newlyUnlocked[], allUnlockedIds[], usedLLM, pressure, emotion, stageReveals[], pressureHint? }
一次呼叫完成三個判定（線索觸發 + 情緒評估 + 階段推進），省 LLM 呼叫。
主函數：checkCluesWithAgent(provider, caseData, question, reply, alreadyUnlockedIds, evidenceProgress?)
