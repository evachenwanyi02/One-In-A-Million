Hermes — B2B 自動化外展系統
項目全景解讀

Mad Mad Group · 2025 Summer Internship · 三人團隊全棧項目

這份文件不講代碼語法，而是講清楚整個系統在做什麼、怎麼做、遇到了什麼難題、怎麼解決。讀完之後你能向任何人（面試官、客戶、技術主管）用三分鐘講明白這個項目的價值和架構。

第一部分：這個系統解決什麼問題

MAD MAD Group 是一家香港公司，需要大量開發 B2B 客戶。傳統做法是人工在 Google Maps 上找商家、手動抄聯繫方式、逐一撰寫開發信、用 Gmail 一封一封寄出。一個銷售一天最多處理 20 家。

Hermes 把這條流水線全自動化：

  1. 自動發現 — 用瀏覽器代理（CUA）在 Google Maps 上按關鍵字 + 地區搜索商家
  2. 自動豐富 — 訪問每家公司官網，提取聯繫郵箱、電話、服務範圍
  3. AI 分析 — 用大語言模型分析合作可能性，生成合作建議和切入角度
  4. 自動撰寫 — AI 依據分析結果，為每家公司草擬個性化開發信
  5. 審批發送 — 人工審核通過後，系統通過 SMTP 自動寄出
  6. 回覆追蹤 — 定時檢查回覆，AI 分類回覆意向（有興趣 / 約時間 / 拒絕），推送通知

一個人可以同時跑幾十條流水線，一天觸達上百家潛在客戶。系統保留「人工審批」環節，確保不會發出不當郵件。

關鍵詞：Lead Scraper CMS、B2B 自動化外展（Automated Outreach）、人機協作（Human-in-the-Loop）

最核心的架構決策：解耦編排

整個系統最重要的設計是：NestJS 永遠不做重活。所有 AI 推理、瀏覽器自動化、郵件發送都委託給外部的 Hermes Agent（Python Worker）。NestJS 只負責編排——往 MongoDB 的任務隊列裡放任務，Agent 自己認領、執行、回報結果。

NestJS 是編排器，不執行重活。Agent 通過 MongoDB 任務隊列與後端解耦，可獨立擴展。

這個設計的好處：NestJS 響應請求始終很快（不阻塞），Agent 可以在不同機器上部署多個實例來水平擴展，且 Agent 崩潰不會影響 Web 服務器。

第三部分：五階段流水線 — 系統核心

當用戶在搜索頁輸入「瑜伽館 · 中環 · 找 5 家」並點擊搜索，系統啟動一條流水線（Campaign），經過五個階段自動完成整個外展流程：

整條流水線由事件驅動自動推進，前端通過 SSE 實時顯示進度。

Pipeline 的關鍵機制

Campaign 追蹤：每次搜索建立一條 Campaign 記錄（編號 CAMP-XXXXXXXX），記錄目標數量和各階段已完成數量。前端通過 SSE 的 pipeline_progress 事件實時更新搜索圈的進度百分比。

任務隊列的原子操作：Agent 認領任務用 MongoDB 的 findOneAndUpdate，保證在多 Agent 環境下不會重複認領。卡死超過 15 分鐘的任務由定時任務自動回收重排。

郵件審批狀態機：郵件有嚴格的狀態流轉：pending → approved → sent，或 pending → rejected。代碼裡用明確的狀態轉移表（State Machine）限制非法跳轉，例如不能從 rejected 直接變成 sent。

第四部分：前端——十個頁面的設計邏輯

儀表板（Dashboard）

四張 KPI 卡片（待審核郵件、新回覆、待約會議、待跟進），搭配漏斗進度條（線索→已聯絡→已回覆→約會議→累計發信）和甜甜圈圖（回覆分類統計）。所有指標從 Leads 和 Email Queue 的原始數據在前端即時計算——不依賴後端額外接口。

潛在客戶搜索（Search）

核心頁面。居中的動態光暈搜索欄，輸入關鍵字 + 香港 19 區 + 目標數量後觸發流水線。搜索進行中時，搜索欄變形為 144px 的進度圈，SVG stroke-dasharray 動畫顯示完成百分比，下方即時滾動日誌流。搜索完畢後展示結果卡片，附帶模擬 Google Maps 抓取過程的瀏覽器預覽動畫。

潛在客戶池（Leads）

系統最複雜的頁面（2800+ 行）。上方 KPI 統計條 + 主/子標籤頁系統。主標籤：準備中、等待回覆、已回覆。子標籤進一步細分：有興趣、約時間、有問題、沒興趣。表格按日期分組（今日/昨日/更早），分組標頭可摺疊（帶平滑 CSS 過渡動畫）。

點擊列表項展開全屏詳情面板：公司資訊卡（帶彩色圖標）、客戶歷程時間線、標籤、回覆分析，右側是該公司所有關聯郵件的完整記錄。每封郵件以紙張質感卡片呈現（帶摺角），可單獨滾動，支持直接審批/拒絕/發送。

發件箱（Email Queue）

左側是按狀態分組的文件夾（全部/待處理/已核准/已拒絕/已發送/失敗），帶徽標計數。中間是郵件列表，支持多選批量操作。右側是郵件詳情。第二個標籤頁是郵件模板編輯器——三欄布局，帶變量插入和即時預覽。

任務看板（Tasks）

四列看板：待處理、處理中、已完成、失敗。每張卡片顯示任務類型標籤（S1-S4）、優先級、執行 Agent。點擊展開工作流面板，按步驟顯示執行時間線和進度條。

AI Agent 管理

最具視覺特色的頁面。左半部是等距像素藝術場景——維多利亞港天際線、摩天輪動畫、辦公室裡打字的開發者、睡覺的貓、搖尾巴的狗，根據當前時間切換日/夜場景。右半部顯示四個 Agent 的環形進度圖（Lead Scraper / Lead Analyzer / Email Drafter / Email Sender）、活動記錄和配置表。

第五部分：技術難點與解決方案

難點一：瀏覽器自動化 vs 反爬蟲

問題：Google Maps 有嚴格的反爬蟲機制，傳統的 HTTP 爬蟲根本無法使用——頁面是 JavaScript 渲染的，且會檢測自動化行為。

解決：使用 CUA（Computer Use Agent）——不是在代碼層面操控 DOM，而是像真人一樣操控滑鼠和鍵盤，在真實瀏覽器裡模擬搜索、滾動、點擊商家卡片、複製信息。這比 Puppeteer/Playwright 更難被檢測。

難點二：NestJS 與 Python 管線共存

問題：項目從 Python 管線遷移而來，MongoDB 裡已有大量 Python 寫入的數據——snake_case 命名、字符串格式的時間戳、status 為 null 代表新記錄。NestJS 的 Mongoose 模型預設會拒絕未定義的欄位。

解決：Mongoose Schema 設置 strict: false，保留 Python 寫入的所有欄位。公司名稱規範化函數同時處理中文法律後綴（有限公司、集團）和英文後綴（Ltd, Inc, Corp），確保去重邏輯跨語言一致。

難點三：流水線的可靠串聯

問題：五階段流水線，每階段可能失敗或超時。S1 找到 5 家公司，每家要獨立走完 S2→S3→S4。如何保證不丟失、不重複？

解決：用 RxJS Subject 做進程內事件總線。任務完成時發出 TaskEvents.completed$ 事件，Hermes 模塊訂閱後判斷完成的是哪個階段，自動排入下一階段任務。Campaign 記錄用 MongoDB 的 $inc 原子操作更新計數器。卡死任務由 Cron Job 每 10 分鐘掃描回收。

難點四：前端實時性

問題：流水線可能跑幾分鐘到幾十分鐘，用戶需要實時看到進度。輪詢太慢且浪費帶寬。

解決：雙層實時策略——主路徑是 SSE（Server-Sent Events），後端推送 lead_update、email_update、pipeline_progress 等六種事件，前端收到後用 invalidateQueries 刷新對應的 React Query 緩存。備用路徑是 120 秒輪詢，保證 SSE 斷線時不丟數據。Agent 在 NestJS 進程外執行，通過 POST /api/sse/notify 向 SSE 總線推送事件。

難點五：2800 行的超大組件

問題：Leads 頁面集中了列表、篩選、分組、排序、詳情面板、郵件操作等大量功能。單文件 2800+ 行，維護困難。

解決：在單文件內用清晰的分區策略——styled-components 定義在文件前半部分，子組件（如 LeadEmails）用獨立的函數組件定義在主組件之前。用自定義 Hook（useDialog）抽離彈窗邏輯。CSS 動畫（keyframes）集中定義在文件頂部。面對這個量級的代碼，清晰的分區比強行拆文件更實用——因為拆分會增加 props 傳遞的複雜度。

難點六：表格行的平滑摺疊動畫

問題：HTML 的 <tr> 不支持直接動畫 height。想讓表格分組摺疊時有平滑的收縮效果而不是硬切。

解決：不隱藏行，而是保留行的 DOM 存在。通過在 <td> 上做 CSS transition——同時過渡 padding、max-height、opacity 和 border-color，實現從可見到零高度的平滑收縮。用 $collapsed prop 控制狀態切換。

難點七：暗色主題全局一致性

問題：系統支持亮色/暗色切換，但有些顏色（如郵件卡片的紙張色、詳情面板標頭色）不能簡單反轉——需要單獨設計。

解決：用 useTheme() 的 styledTheme.mode === 'dark' 判斷當前模式，為特殊元素（紙張背景、摺角漸變、標頭底色等）提供獨立的暗色方案。主題狀態持久化到 localStorage，刷新後保持選擇。

第六部分：面試怎麼講這個項目

30 秒版本

「我在三人團隊裡負責前端，項目是一個 B2B 自動化外展系統。它用瀏覽器代理自動在 Google Maps 上找商家，用 AI 分析合作可能性並撰寫個性化開發信，通過人工審批後自動發送，還能追蹤回覆意向。技術上用 React + TypeScript + styled-components 做前端，NestJS + MongoDB 做後端，通過任務隊列解耦 AI 推理和瀏覽器自動化。」

三分鐘版本——可以展開的技術點

架構層面：講解耦編排——NestJS 永不做重活，所有 AI 和瀏覽器操作通過 MongoDB 任務隊列委託給外部 Agent，用 RxJS 事件驅動串聯五階段流水線。這讓 Web 服務器始終響應快速，且 Agent 可獨立擴展。

前端層面：講 TanStack Query + SSE 的實時數據策略——不用 Redux，用 React Query 管理服務端狀態，通過 SSE 事件自動刷新緩存。講 styled-components 的主題系統如何實現亮/暗色切換。講 i18next 如何覆蓋 250+ 翻譯鍵實現三語支持。

難點層面：講 CUA 為什麼比 Puppeteer 更適合反爬蟲場景。講 Python 遺留數據的兼容策略。講超大組件的分區管理哲學。
